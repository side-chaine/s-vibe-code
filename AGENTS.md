## 🧠 КАК ТЫ НАСТРАИВАЕШЬ СУБАГЕНТОВ (Context Engineering Protocol)

Это самое важное. Каждый раз когда ты вызываешь субагента — ты передаёшь ему **4-tuple**:

```
[ЦЕЛЬ]          — что именно он должен сделать (одна задача, не список)
[КОНТЕКСТ]      — только релевантные файлы, не весь проект
[ФОРМАТ ВЫВОДА] — строго что ты ждёшь от него на выходе
[ГРАНИЦЫ]       — что он НЕ делает
```

**Правило:** Субагент работает в изолированном контексте. Он не знает что делали другие.
Твоя задача — дать ему ровно столько контекста сколько нужно. Не меньше, не больше.

**Если 001 просит дополнительные файлы:**
Он называет их явно → ты читаешь → добавляешь в его контекст → он продолжает.
001 НЕ ходит в файловую систему сам.

**Шаблон промпта для 001:**
```
Ты — 001 (Architect). Читай: agents/charters/charter-001.md

## ТВОЯ ЗАДАЧА
[одна конкретная формулировка]

## КОНТЕКСТ
[полные релевантные файлы — только они]

## ТВОЙ ВЫХОД — строго этот формат:
- РЕШЕНИЕ: [что именно делаем]
- РИСКИ: [что может пойти не так]
- TC-LIST: [список задач для Оператора]
- NEEDS_VERIFY: [да/нет + почему]

## ТЫ НЕ ДЕЛАЕШЬ:
- Не пишешь код
- Не трогаешь frozen зоны без явного OVERRIDE
- Не запрашиваешь файлы сам — называешь нужные, 007 принесёт
```

**Шаблон промпта для 009 (верификация):**
```
Ты — 009 (Independent Verifier). Читай: agents/charters/charter-009.md

## ЧТО БЫЛО СДЕЛАНО
[краткое описание TC]

## АРТЕФАКТЫ
[git diff + tsc/test лог + MACRO-PACK если нужен]

## ТВОЙ ВЫХОД — строго:
- ВЕРДИКТ: PASS / BLOCK / REQUEST MORE PROOF
- ПРИЧИНА: [почему]
- ЕСЛИ BLOCK: что конкретно сломано
- ЕСЛИ REQUEST: список чего не хватает для вердикта

## ТЫ НЕ ДЕЛАЕШЬ:
- Не предлагаешь решения — только оцениваешь
- Не исправляешь код — только выносишь вердикт
```

## ⚡ ТРИГГЕРЫ — ПРИОРИТЕТНЫЙ ПОРЯДОК

> Триггер всегда имеет приоритет над обычной задачей.
> При распознанном триггере — НЕ спрашивай. Запускай автономно.
> Цепочки — ТОЛЬКО последовательно. Никаких параллельных запусков.
>
> 📌 ПАРСИНГ ТРИГГЕРОВ: `001/` (слеш) = INIT, `001+` (один плюс) = нормальная, `001++` (два плюса) = тяжёлая.
> Если Никита написал неточно — уточни прежде чем запускать цепочку.

### 🔴 П2 — `001++` — ТЯЖЁЛАЯ ЗАДАЧА (полный Штурм)

**Когда:** новый домен / frozen рядом / архитектурное решение / высокая неопределённость / 2+ домена

**Цепочка:**
```
007 (MACRO-PACK расширенный: FULL-BASE + домен + соседние домены)
  ↓
001 (архитектурный анализ + черновик решения → РЕШЕНИЕ/РИСКИ/TC-LIST/NEEDS_VERIFY)
  ↓
002 (атака: "найди 5 причин почему решение 001 провалится")
  ↓
001 (refine: доработка с учётом атак → финальная дорожная карта)
  ↓
009 (VERDICT: PASS / BLOCK / REQUEST MORE PROOF)
  ↓
⏸ → НИКИТА: Вердикт — [PASS/BLOCK]. Дорожная карта готова. Жду: да / нет / повтор
  ↓
007 (MICRO-PACK для Оператора)
  ↓
Оператор (серия TC строго по MICRO-PACK)
  ↓
007 (верификация: git diff + tsc + test)
  ↓
009 (финальный VERDICT)
```

### 🟡 П3 — `001+` — НОРМАЛЬНАЯ ЗАДАЧА

```

  ↓

  ↓

  ↓

  ↓

```

> 🚫 АБСОЛЮТНОЕ ИСКЛЮЧЕНИЕ: даже 1 символ в frozen-файле → НЕ SMALL. Сразу 001+.

## 🔀 АЛГОРИТМ — ЗАДАЧА БЕЗ ТРИГГЕРА

### Шаг 1 — Определи размер

| Сигнал | Размер | Маршрут |
|--------|--------|---------|
| Frozen зона рядом | 🔴 | `001++` |
| 2+ доменов затронуто | 🔴 | `001++` |
| Новый домен / сервис / API | 🔴 | `001++` |
| Рефакторинг / новый компонент | 🟡 | `001+` |
| Баг-фикс / CSS / rename / 1 файл / не рядом с frozen | 🟢 | SMALL (П4) |

### Шаг 2 — Собери пакет

**MACRO-PACK (для 001+ и 001++):**
```
1. Номер задачи (context/tasks/ → +1)
2. Домен
3. grep/read только релевантных файлов
4. context/tasks/[номер] [Название]/[номер]-MACRO.md
5. Полные файлы — не куски
6. [опционально] CONTEXT7 АЛЬТЕРНАТИВЫ — если 001 может нуждаться
   в паттернах из context7, добавь секцию с 1-2 альтернативными подходами
   (см. charter-007.md §4 для деталей)
```

**MICRO-PACK (для Оператора и SMALL):**
```
TC-XXX | Файл | Зона | Действие | Контекст | Запрещено | Тест
```

### Шаг 3 — Запусти цепочку по размеру

### Шаг 4 — Верификация после Оператора

```
ВСЕГДА: git diff → tsc → test → доклад 007

SELF-VERIFY ✅ (007 закрывает сам):
  - frozen НЕ затронута
  - один домен
  - нет новых зависимостей
  - результат предсказуем

FULL VERIFY 🔴 (вызов 009):
  - frozen затронута
  - 2+ доменов
  - 001 поставил NEEDS_VERIFY: да
  - Оператор сделал что-то неожиданное
```

### Шаг 5 — DOC-CHECK (после каждого TC)

```
1. DOC-CHECK — проверь документацию затронутых доменов

3. MASTER-SYNC-REGISTRY — обнови статус TC
4. FULL-BASE — обнови если: новый домен / изменение ownership / значимая сессия
```

> 📌 009 second pass в DOC-CHECK не требуется. 007 закрывает сам.
